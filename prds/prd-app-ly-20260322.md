## Executive Summary

app-ly is a locally-hosted, AI-powered resume tailoring tool. Given a base resume or skills list and a target job listing, it uses a two-agent Anthropic pipeline to produce a tailored, ATS-optimized resume exported as DOCX (with optional PDF/ODT via LibreOffice). An optional feedback loop allows the tailoring agent to refine its output using the review agent's critique. The tool is designed for individual job seekers who want to maximize interview conversion rates without manually rewriting their resume for every application. All PII (contact details) is handled exclusively client-side and never sent to the LLM.

---

## Product Vision

### Problem Statement

Job seekers must tailor their resume to each application to pass Applicant Tracking Systems (ATS) and capture hiring manager attention — but doing this manually is time-consuming, inconsistent, and easy to get wrong. Most applicants either submit a generic resume (low match rate) or spend 30–60 minutes per application rewriting content (unsustainable at scale).

### Solution Overview

app-ly automates resume tailoring via an AI pipeline with three stages:
1. A **tailoring agent** that extracts keywords and requirements from the job listing, selects the most relevant experiences and skills from the user's input, and rewrites them into a polished, ATS-optimized resume body.
2. A **review agent** acting as a hiring manager / ATS critic that scores the output and provides structured feedback.
3. An optional **refinement pass** where the tailoring agent uses the review agent's critique to improve the resume before export.

Contact details (name, email, phone, etc.) are collected in dedicated UI fields and injected into the final document locally — they are never included in any LLM prompt.

### Value Proposition

- Reduces per-application resume tailoring from 30–60 minutes to under 5 minutes
- Applies consistent ATS best practices on every resume without the user needing to know them
- Provides an independent AI review pass and optional refinement loop
- PII stays off the wire — contact info never leaves the machine in an LLM request
- Fully local — no data stored, no third-party SaaS, user controls their own API key

## User Stories

- As a job seeker, I want to paste my resume text and a job listing so that I receive a tailored resume without manually editing it.
- As a job seeker, I want to paste a bullet-point skills list instead of a full resume so that I can still get a complete tailored resume even without a polished base document.
- As a job seeker, I want the tailored resume exported as DOCX so that I can submit it to employers, with PDF/ODT also available if LibreOffice is installed.
- As a job seeker, I want an AI review of the tailored resume so that I can see weaknesses and a confidence score before I apply.
- As a job seeker, I want the option to have the AI refine the resume based on its own review so that I get a stronger result without extra effort.
- As a job seeker, I want my personal contact details kept out of LLM requests so that my PII is not transmitted to a third-party API.
- As a job seeker, I want to provide my own Anthropic API key so that I control my own usage and costs.
- As a job seeker, I want to access the app from my desktop browser on my local network so that I don't have to work directly on my server.

## Functional Requirements

### 1. Input Collection

**Description:** A Streamlit form that accepts all user inputs needed to generate a tailored resume.

**Contact Fields (PII — never sent to LLM):**
- Full name (text input, required)
- Email address (text input, required)
- Phone number (text input, optional)
- Location / City, State (text input, optional)
- LinkedIn URL (text input, optional)
- GitHub URL (text input, optional)

**Content Fields (sent to LLM):**
- Anthropic API key (text input, masked, required) — pre-filled from the `ANTHROPIC_API_KEY` environment variable if present; field is hidden when the env var is set
- Resume / Skills input (multi-line text area, required) — plain text resume body or bullet-point skills/experience list; placeholder text explicitly instructs the user to omit their name, email, and phone
- Job listing (multi-line text area, required) — full job description text
- Target role (text input, optional) — e.g., "Senior Software Engineer"; gives the tailoring agent an explicit anchor for the summary and skills sections beyond inferring the role from the JD

**Options:**
- Page limit (optional numeric input, default: no limit) — if set, instructs the tailoring agent to constrain content to fit within the specified number of pages and instructs the review agent to flag any content likely to exceed it

**Acceptance Criteria:**
- Name, email, API key, resume/skills, and job listing must all be non-empty before the generate action is enabled
- API key is not stored anywhere (stateless; only held in session memory for the duration of the request)
- Contact fields are never included in any prompt sent to the Anthropic API
- Placeholder text in the resume/skills field instructs the user to omit contact information
- Clear placeholder text guides the user on what to paste in each field

**Priority:** Must Have

---

### 2. Tailoring Agent

**Description:** An Anthropic API call that produces a structured, tailored resume body from the user's content inputs. Contact information is explicitly excluded from the LLM prompt and output schema.

**Inputs to LLM prompt:**
- Resume/skills text (content only, no PII)
- Job listing text
- Target role (if provided)
- Page limit (if set)

**Behavior:**
- Extracts required and preferred skills/keywords from the job listing
- Selects and prioritizes the most relevant experiences, skills, and achievements from the user's input
- Rewrites bullet points to use strong action verbs, past tense, and quantified impact where possible
- Uses target role (if provided) to anchor the summary and skills framing
- Ensures section headers match ATS-standard naming conventions (e.g., "Work Experience", "Skills", "Education")
- Avoids tables, columns, graphics, or non-standard formatting in the content structure
- If a page limit is set, trims content aggressively to the most impactful experiences and skills to respect the constraint; the system prompt explicitly states the target page count
- Outputs a structured object via Anthropic tool use (enforced schema — no prompt-based JSON parsing or retry logic required)
- Includes a `rationale` field: a brief plain-text explanation of what was prioritized, what was omitted, and why

**Model:** `claude-sonnet-4-6`

**Acceptance Criteria:**
- Output is delivered via tool use and is always a valid, schema-conformant object — no parse failures
- All standard resume sections are present where data exists: Summary, Work Experience, Skills, Education
- Keywords from the job listing appear naturally in the resume content
- No content is fabricated — all output is derived from the user's provided input
- Output contains no contact or PII fields
- `rationale` field is present and human-readable

**Priority:** Must Have

---

### 3. Review Agent

**Description:** A second Anthropic API call that critiques the tailored resume body from the perspective of an ATS system and a hiring manager.

**Inputs to LLM prompt:**
- ResumeBodyJSON (no contact fields)
- Job listing text
- Target role (if provided)
- Page limit (if set)

**Behavior:**
- Reviews keyword coverage against the job listing
- Flags missing required qualifications, weak bullet points, or thin sections
- Checks for ATS formatting issues in the content
- If a page limit is set, flags content that is likely to exceed it given the fixed template's font size and margins
- Produces an overall confidence score (0–100) and prioritized lists of strengths, concerns, and actionable suggestions
- Output delivered via Anthropic tool use (enforced schema — no parse failures)

**Model:** `claude-sonnet-4-6`

**Acceptance Criteria:**
- Output is delivered via tool use and is always a valid, schema-conformant object
- ReviewJSON contains: score (integer 0–100), strengths (list), concerns (list), suggestions (list)
- Review is displayed in the Streamlit UI alongside the resume plain-text preview
- Review references specific sections or content from the generated resume

**Priority:** Must Have

---

### 4. Optional Refinement Loop

**Description:** After the review is displayed, the user may optionally trigger a refinement pass where the tailoring agent receives its own output plus the review critique and produces an improved resume.

**Behavior:**
- A "Refine Resume" button appears alongside the review results
- On click, the tailoring agent is called again with: original resume/skills input, job listing, target role (if set), current ResumeBodyJSON, ReviewJSON (suggestions and concerns), and page limit (if set)
- The agent produces a new ResumeBodyJSON incorporating the feedback
- The review agent then re-evaluates the refined resume and displays an updated ReviewJSON
- Limited to a maximum of **2 refinement iterations** per session to prevent runaway API usage
- The iteration count and a "Max refinements reached" message are shown when the limit is hit

**Acceptance Criteria:**
- Refinement is opt-in — the user must click to trigger it
- Each refinement pass runs both agents (tailor → review) and updates both the preview and review panel
- Iteration counter is visible in the UI (e.g., "Refinement 1 of 2")
- Export buttons always reflect the most recent version of the resume
- PII fields are still never included in any refinement prompt

**Priority:** Must Have

---

### 5. Resume Export

**Description:** Generation and download of the tailored resume from the final ResumeBodyJSON, with contact fields injected locally at render time.

**PII injection:** Contact fields collected in the UI (name, email, phone, location, LinkedIn, GitHub) are merged into the document during rendering — after all LLM calls have completed. They are not stored in any intermediate JSON written to disk.

**Formats:**
| Format | Method | Availability |
|--------|--------|-------------|
| DOCX | `python-docx` rendering against fixed template | Always available |
| PDF | LibreOffice CLI (`soffice --headless --convert-to pdf`) | Only if LibreOffice found on PATH |
| ODT | LibreOffice CLI (`soffice --headless --convert-to odt`) | Only if LibreOffice found on PATH |

**LibreOffice unavailability behavior:** If LibreOffice is not found on PATH at app startup, a persistent info banner is shown in the UI: "PDF and ODT export unavailable — LibreOffice not found. Install LibreOffice and restart the app to enable these formats." The DOCX download button is always present and unaffected.

**Template Specification (Fixed):**
- Single-column layout
- Font: Calibri (or Liberation Sans as open-source fallback)
- Name: 18pt bold, centered
- Contact line: 10pt, centered, pipe-separated (email | phone | location | LinkedIn | GitHub)
- Section headers: 12pt bold, left-aligned, with a thin bottom border
- Body text: 11pt, left-aligned
- Bullet points: standard indented list, no custom symbols
- Margins: 0.75in all sides
- No photos, logos, tables, text boxes, or columns
- Page size: US Letter

**Acceptance Criteria:**
- DOCX download button is always present after generation
- PDF and ODT download buttons appear only if LibreOffice is available
- Downloaded DOCX renders correctly in LibreOffice Writer and Microsoft Word
- Contact fields appear in the rendered document but were never part of any LLM prompt
- Resume content matches the final ResumeBodyJSON exactly (no rendering artifacts)

**Priority:** Must Have

### 6. UI Layout and Flow

**Description:** The end-to-end Streamlit user experience.

**Flow:**
1. User opens the app in their browser
2. User fills in contact fields (name, email, etc.)
3. User enters API key, resume/skills body text, and job listing text
4. User clicks "Generate Tailored Resume"
5. App shows step-by-step progress: "Tailoring resume..." → "Running review..."
6. App displays:
   - Plain-text resume preview (rendered from ResumeBodyJSON, contact fields shown at top)
   - Collapsible "Why this resume?" section showing the tailoring agent's rationale (what was prioritized, what was omitted, and why)
   - AI review panel: score badge, strengths, concerns, suggestions
   - "Refine Resume" button (if under iteration limit)
   - Download buttons: DOCX always; PDF and ODT if LibreOffice available
7. User may click "Refine Resume" to trigger a refinement pass (updates preview, review, and downloads)
8. User can edit any input and regenerate from scratch without refreshing the page

**Acceptance Criteria:**
- Page does not require a full refresh between generation or refinement runs
- Progress state is communicated clearly per pipeline step (spinner + status text)
- Error states are surfaced clearly: invalid API key, API failure, malformed JSON, LibreOffice not found
- Refinement iteration counter is visible when refinement has been used

**Priority:** Must Have

## Non-Functional Requirements

### Performance
- End-to-end generation (both agent calls + DOCX export) completes in under 60 seconds under normal Anthropic API latency
- Each refinement pass completes in under 60 seconds
- LibreOffice conversion completes in under 10 seconds per format

### Security / Privacy
- API key is never written to disk, logged, or persisted beyond the active Streamlit session
- PII (name, email, phone, location, LinkedIn, GitHub) is never included in any LLM prompt
- No user data leaves the local machine except via the user's own Anthropic API calls (which contain only resume body content and job listing)
- App binds to localhost or local network interface only — not exposed to the public internet
- A notice is shown in the UI: "Your resume content and job listing are sent to Anthropic's API. Contact details are never included in these requests."

### Platform Support
- Runs on Linux and macOS (primary: Linux server, secondary: macOS dev)
- Accessible from any modern browser on the local network (Chrome, Firefox, Safari)
- LibreOffice is an optional system dependency for PDF/ODT; app functions without it

### Reliability
- Graceful error messages if the Anthropic API returns an error or rate limit
- Graceful handling of malformed JSON from the LLM: retry once with a corrective prompt; surface a clear error if retry also fails
- Graceful error message if LibreOffice is not found on PATH
- No unhandled exceptions that crash the Streamlit process

## Risks and Mitigation

| Risk | Mitigation |
|------|------------|
| LLM fabricates experience or credentials | System prompt explicitly instructs the agent to only use content from the user's input; review agent also checks for plausibility |
| User accidentally includes PII in resume/skills text area | Placeholder text and a UI warning advise against it; PII in that field would still reach the LLM — user is informed this is their responsibility |
| LibreOffice not installed on host | Detected at startup; DOCX download always works; persistent info banner explains missing formats |
| Anthropic API key invalid or rate-limited | Catch API errors and surface a clear message; do not crash the app |
| JSON output from LLM is malformed | Wrap parse step in error handling; retry once with a corrective prompt if parse fails |
| Resume content exceeds page limit | If a page limit is set, the tailoring agent trims aggressively and the review agent flags overruns; without a limit, template and font sizing maximize content density |
| Refinement loop degrades quality | Both pre- and post-refinement scores are shown so the user can compare; user chooses whether to export either version |

---

## Resolved Design Decisions

1. **Contact information** — Collected in dedicated UI fields. Never included in LLM prompts. Injected into the document locally at render time.
2. **Resume preview** — Plain text rendered from ResumeBodyJSON in Streamlit.
3. **LibreOffice fallback** — DOCX export always available. PDF/ODT buttons appear only when LibreOffice is found. A persistent info banner explains if they are unavailable.
